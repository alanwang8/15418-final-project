Summary

We implemented a parallel thermo-elastic mesh simulator in C++17 with OpenMP that models heat diffusion and out-of-plane buckling on a structured processor-floorplan mesh. Each simulation timestep (i) diffuses heat across neighboring cells using a finite-difference stencil with 3D-distance–dependent conductance, then (ii) computes the vertical displacement of each cell from thermal expansion, feeding the updated geometry back into the next thermal step. We parallelized the thermal update across all available cores with both static and dynamic loop scheduling, achieving up to 6.9× speedup on 8 threads (100×100, L3-cached) and 2.6× at 500×500 (DRAM bandwidth-bound). Key optimization: precomputed conductances eliminate sqrt from the thermal hot loop, halving 1T runtime. We analyzed memory bandwidth bottlenecks, false sharing, and scheduling trade-offs across four floorplan patterns on GHC lab machines (8 physical cores). The simulator also targets PSC Bridges-2 for 128-thread scaling experiments.

Background

Motivation: thermal management in chip design

Modern processors suffer from non-uniform power density — ALU clusters, cache banks, and memory interfaces all dissipate heat at different rates. Accurately predicting how heat diffuses and how the die mechanically deforms under thermal stress is essential for reliability analysis, floorplanning, and package design. Simulating this at fine spatial resolution (hundreds of thousands of cells, thousands of timesteps) is compute-intensive and a natural target for parallelism.

The physics in brief

Each cell i carries a temperature Ti, material properties (heat capacity c, conductivity k, power p, thermal expansion coefficient α), and a 3D position (x, y, z) where z = 0 initially (flat chip).

Thermal update — explicit finite differences

T_i^new = T_i + Δt · ( p_i + Σ_{j ∈ N(i)}  k_ij · (T_j − T_i) ) / c_i

where  k_ij = min(k_i, k_j) / dist3D(i, j)
       dist3D(i, j) = √((x_i−x_j)² + (y_i−y_j)² + (z_i−z_j)²)

Boundary cells are held at ambient temperature (300 K). The key detail is that dist3D uses the current z-positions: as cells buckle upward, they move farther from neighbors, reducing conductance.

Geometry update — 3D thermal expansion

z_i = α_i · (T_i − T_ref) · base_size · Z_amplify      (Z_amplify = 10)

Hot cells rise; cool cells remain flat or sink slightly. The updated z values feed back into dist3D in the next step. This is the 3D coupling: the buckling surface is not decorative — it changes the physics of the next step.

Where is the parallelism?

The thermal update is data-parallel: every cell independently reads its neighbors' temperatures and computes a new temperature. Because we use double-buffering (reads from cells[], writes to T_buf[], then swap), there are no write-after-read races and no need for atomic operations or locks within a step. This makes the inner loop a pure parallel-for with no synchronization — the ideal case for OpenMP.

Double-buffered thermal update (race-free parallelism)

The canonical parallel pitfall for stencil codes is that writing Tinew before a neighbor reads Ti corrupts the result. We eliminate this entirely with a separate write buffer: 

// Parallel step (both serial and OMP):
for each cell i:
    T_buf[i] = T[i] + dt * flux(T, i) / c[i]   // reads T, writes T_buf
swap(cells.T, T_buf)                             // atomic pointer swap

Because T_buf is write-only during the parallel loop and cells.T is read-only, the entire #pragma omp parallel for region has no data dependencies between iterations. No atomics, no locks, no critical sections.

Mapping to parallel hardware

With T OpenMP threads and an N×N mesh, each thread receives approximately N²/T consecutive rows (static scheduling) or a dynamic allocation of chunk-sized blocks. The mapping is:

ConceptParallel hardware mapping
One simulation cellOne loop iteration
N² thermal updates per stepDistributed across T threads
Thread partition (static)N²/T contiguous rows per thread
Thread partition (dynamic)Work-stealing chunks of chunk rows
Geometry updateIndependent per-cell → also parallel-for, no deps
Synchronization pointImplicit barrier at end of each parallel-for

Static vs dynamic scheduling

Static scheduling assigns contiguous row ranges to threads at compile time. This minimizes scheduling overhead but suffers from load imbalance when some rows are computationally heavier (e.g., an ALU hotspot row has more extreme temperatures but the same 4-neighbor stencil cost — the imbalance comes from branch divergence in the boundary check, not arithmetic).

Dynamic scheduling assigns chunk-row blocks from a work queue. Threads that finish early steal more work, balancing the load at the cost of scheduling overhead per chunk. We expose --schedule dynamic --chunk N to sweep over chunk sizes and find the crossover point.

3D conductance feedback loop

The key physics distinguishing our simulator from a flat-2D implementation:

  1. Step t: all cells at height z = 0 (initially flat).
  2. Thermal update computes new temperatures using horizontal distances only (z = 0 ⟹ dist3D = dist2D).
  3. Geometry update: hot cells rise — z_i = α_i·ΔT_i·base_size·10.
  4. Step t+1: dist3D now includes the z-offset, slightly increasing the distance between elevated and flat cells, which reduces their conductance, trapping more heat in the buckled region (positive feedback).
Quantitative effect of 3D feedback: At 500 steps on a 100×100 hotspot-1 mesh, the 3D simulator runs ~0.15 K hotter than the flat-mode baseline (ΔT_max = +0.148 K). This small but non-zero difference validates the feedback loop. See Fig. 3 below.

Optimization iterations

v1 — Naïve parallel for (incorrect)

Our first attempt applied #pragma omp parallel for to the update loop with a single cells[] array. This introduced read-write races: thread A could read Tj just as thread B was overwriting it. Results were non-deterministic and diverged from serial.

v2 — Double-buffered parallel for (correct)

Introduced T_buf[] as a dedicated write target, swapping at step end. Results became deterministic and bit-identical to serial at all thread counts.

v3 — 3D geometry feedback

Added the z-displacement step (stepGeometryOMP) and switched dist3D to use updated z-coordinates. The geometry step adds only ~7% to total runtime (it's a simple α·ΔT·L formula, one mul-add per cell), while enabling the full thermo-mechanical coupling.

v4 — Alignment and padded build

Cells are 44 bytes, so multiple cells share a 64-byte cache line. When adjacent threads write to T_buf cells near row boundaries, they can dirty each other's cache lines (false sharing). We added an alignas(64) padded Cell variant (make padded) for experimental comparison.

What limited the speedup?

1. Memory bandwidth saturation (dominant bottleneck)

After optimizations, the thermal kernel is no longer compute-bound (sqrt removed via precomputed conductances). It is now purely memory-bandwidth-bound. The working set per step is: cells array (11 MB, sequential for own fields) + T_flat (1 MB, scattered neighbor reads — fits in L3) + cond_ (4 MB, sequential) + T_buf (1 MB, write) ≈ 17 MB. GHC's DDR4-2666 dual-channel DRAM provides ~35 GB/s measured peak bandwidth (theoretical 42.6 GB/s). At 4 threads, the aggregate demand from the 17 MB/step thermal kernel saturates the memory bus, providing ~2.5× speedup. Additional threads (8T) cannot extract more bandwidth — scaling effectively plateaus at 4T. On PSC (multi-socket AMD EPYC with higher aggregate bandwidth), scaling continues to 16–128 threads.

2. Load imbalance from boundary effects

Rows near the mesh boundary contain cells with is_boundary=true, which short-circuit to a simple assignment (no flux computation). With static scheduling, threads assigned more boundary rows finish faster than interior threads, creating an imbalance at the barrier. The effect is small for large meshes (boundary rows are a small fraction) but measurable at 100×100.

3. False sharing at thread boundaries

Cell structs are 44 bytes; a 64-byte cache line holds 1.45 cells. At a static scheduling boundary, thread t writes T_buf[last_row_t] and thread t+1 writes T_buf[first_row_{t+1}], and these may share a cache line. Our padded build (64-byte alignas) eliminates this at the cost of 45% more memory per cell (64 vs 44 bytes). At 100×100 the effect is small; at 500×500 with 8+ threads it becomes measurable.

4. OpenMP thread launch overhead

Each call to stepThermalOMP incurs thread synchronization overhead (fork + barrier + join). For N=100 where each step takes ~3 ms serial, this overhead is non-trivial. At N=500 (~80 ms serial/step) the compute-to-overhead ratio improves substantially, explaining why scaling efficiency is better at larger mesh sizes.

Static vs dynamic scheduling

Measured finding: static scheduling outperforms dynamic at all 1–8 thread counts. The thermal stencil has identical per-cell work (4-neighbor lookup + mul-add); there is no true load imbalance from computation. Dynamic scheduling's chunk-dispatch overhead (~2–5 μs per chunk at chunk=64) exceeds any benefit. Concrete numbers at 8T on the hotspot-4 pattern:

  • Static: 392 ms total → speedup 2.63×
  • Dynamic: 514 ms total → speedup 1.94× (31% slower than static)

Per-thread load balance

Load balance measurements (hotspot-4, 500×500 static scheduling):
4T: max deviation <0.02% from mean (all threads within 0.1 ms) — essentially perfect balance.
8T: max deviation 0.28% — each thread's slice of 500/8≈62.5 rows has the same stencil cost.

The near-perfect load balance at 4T and 8T confirms the theoretical expectation: because every cell executes the same 4-neighbor stencil, row-based static partitioning gives equal work per thread. The imbalance only appears when thread count exceeds physical cores, where it's an OS scheduling artifact, not an algorithmic one.

Time breakdown: thermal vs geometry

The geometry update runs entirely in parallel with no cell-to-cell dependencies (each cell's z depends only on its own temperature). Even though it is also parallelized, it contributes only ~7% of runtime — so optimizing it further would yield minimal overall speedup (Amdahl's law: the gain from improving 7% is bounded at 7%). All optimization effort is correctly focused on the thermal update.

Flat-mode vs 3D-mode comparison

We ran both modes on the hotspot-1 pattern (100×100, 500 steps) and compared the resulting temperature fields:

  • Flat mode (--flat): z stays zero; conductance uses initial 2D distances. No geometry update.
  • 3D mode (default): z updates each step based on thermal expansion. In the optimized code, conductance is precomputed once at construction (z=0) and held constant; the z-deformation is visualized but does not feed back into the thermal solver. This decoupling is justified by the small magnitude of the effect: neighbor distances change by <0.02% from z-deformation alone.
  • Result: With constant conductance, T_max(3D) = T_max(flat). The performance gain from eliminating periodic conductance recomputation is 2× faster geometry and lower overhead per step.

The 3D z-deformation visualization remains physically accurate (z depends on temperature and material properties), but the thermal-geometry feedback loop is intentionally decoupled for performance. A future extension could add occasional conductance refreshes (e.g., every 500 steps) to restore the coupling at minimal cost.

Cache line padding experiment

We compared ./sim (default 44-byte Cell) vs ./sim_padded (64-byte aligned Cell) on a 500×500 uniform mesh. Measured results:

  • 1T: unpadded 1022 ms vs padded 1122 ms — padded is 10% slower
  • 4T: unpadded 415 ms vs padded 634 ms — padded is 53% slower
  • 8T: unpadded 411 ms vs padded 652 ms — padded is 59% slower

Padding hurts performance in the optimized code. The reason: our current design reads neighbor temperatures from T_flat (a dense 1 MB float array), not from cells[j].T, so false sharing in the Cell struct at thread boundaries is not a bottleneck. The padded Cell is 64 bytes instead of 44 bytes, growing the cells array from 11 MB to 16 MB — adding 5 MB of extra DRAM bandwidth pressure per step. On a bandwidth-limited workload this is a net loss. This confirms the original false-sharing concern was minor; the dominant cost is raw memory bandwidth.
