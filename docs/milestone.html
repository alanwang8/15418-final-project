Milestone Report

Dynamic Mesh Simulation for Thermal Diffusion with Parallel Systems

Alan Wang, Allison Chen

CMU 15-418 / 15-618 · Spring 2026 · April 14, 2026

Project Home   Results

Summary

This milestone describes a C++/OpenMP simulator of thermal diffusion coupled to out-of-plane deformation (vertical warping) on a structured processor floorplan mesh. Each timestep applies an explicit thermal update with 3D distance-dependent conductance (kij = min(ki,kj) / dist3D), then updates each cell's z coordinate from thermal expansion so geometry feeds back into the next step. The implementation uses a race-free double-buffered temperature update, static and dynamic OpenMP loop scheduling, and a CLI for threads, mesh size, timestep count, floorplan pattern, and optional field dumps and timing output. We measure strong scaling and produce heatmaps and speedup plots from scripted sweeps.

Technical Approach

Mesh & Data Structures

The floorplan is a grid of square cells in row-major order. Each Cell stores: temperature T, heat capacity c, conductivity k, power p, expansion coefficient α, positions x, y, z, base_size, type (ALU, Cache, Memory, Interconnect), and a boundary flag. Each cell precomputes indices for up to four edge neighbors (no diagonals). Boundary cells are held at ambient temperature (300 K). Four floorplan patterns are supported:

  • Uniform: homogeneous cache-like tiles — uniform T and flat z baseline.
  • Hotspot-1: cache baseline with a central ALU patch (~10% per dimension), high power — single plume and localized warping.
  • Hotspot-4: memory strip on one edge, four ALU clusters at quadrant centers, interconnect corridors between — models a realistic multicore layout.
  • Random: weighted random types with per-cell power jitter; seeded for reproducibility.

Thermal Update

Interior cells use an explicit finite-difference update. All reads come from the current field; writes go to a separate buffer. Buffers swap each step (double-buffering), eliminating write races with no per-cell locks: 

T_i^new = T_i + Δt · (p_i + Σ_{j∈N(i)} k_ij · (T_j − T_i)) / c_i

where k_ij = min(k_i, k_j) / dist3D(i,j).

3D Geometry & Coupling

After each thermal phase, each cell's z-coordinate is updated from thermal expansion (per instructor feedback to support out-of-plane buckling): 

z_i = α_i · (T_i − T_ref) · base_size · Z_amplify

Z_amplify = 10 makes displacement visible in output. Because conductance uses 3D Euclidean distance between cell centers, updated z changes dist3D and therefore k_ij on the following step — creating a genuine thermomechanical feedback loop.

OpenMP Parallelization

Both stepThermal and stepGeometry are parallelized with #pragma omp parallel for. The implicit barrier between phases ensures geometry always sees a consistent post-thermal T. Scheduling is selectable: static (round-robin block assignment, best for uniform workloads) or dynamic with configurable chunk size (for imbalanced hotspot patterns). Thread count is controlled via --threads N; timing uses std::chrono with separate thermal and geometry accumulators.

Preliminary Results

All results below are from a development machine (3-run averages, 100×100 mesh, 200 timesteps, static scheduling). The full GHC sweep (500×500, 1–16 threads) is scheduled for April 22–24.

Correctness Validation

Check: Serial and multi-threaded runs on identical inputs produce the same final temperature field. Double-buffering eliminates write races — threads read from the previous step's buffer and write only to the next buffer before the swap. For Hotspot-1 under benchmark settings, both serial and 8-thread runs converge to Tmax ≈ 320 K and Tmean ≈ 301.5 K.

Uniform Pattern — Baseline Fields

With uniform power and material properties, temperature stays near ambient and z displacement is nearly flat — a sanity check before introducing hotspot gradients.

Hotspot-1 — Central Plume

A single central ALU cluster with 5× power creates a Gaussian-like thermal plume. The corresponding z-displacement map shows a dome-like buckle above the hotspot — the first cells to rise above the chip plane, changing their conductance to neighbors in subsequent steps.

Hotspot-4 — Multicore Floorplan

Four ALU clusters at quadrant centers with interconnect corridors between them model a realistic multicore chip layout. Temperature plumes from each quadrant merge along the interconnect strips; the z-displacement map shows four raised regions corresponding to each hotspot.

Key observations: Uniform workloads achieve near-linear speedup up to 4 threads (4.71×, 95% efficient); the >100% efficiency at 2 threads reflects cache effects — per-thread working sets fit in L1 when the mesh is split, yielding a bonus on top of parallel speedup. Hotspot-1 scales more slowly (6.16× at 8 threads): cells near the hot center have larger temperature gradients and conductance variation, creating mild load imbalance. Thermal update accounts for ~93% of runtime; geometry is lightweight but non-trivial and will grow in relative cost as thermal computation is further optimized.

Concerns & Open Questions

1. Load imbalance and false sharing. Efficiency drops at 8 threads for Hotspot-1 even with equal cell counts per thread. We plan profiling with perf on GHC and layout experiments (cache-line alignment, padding, or SoA for the hot T field).
2. Memory bandwidth at 500×500. The 500×500 mesh (~20 MB for the full cell array) will not fit in L2 cache. We expect lower parallel efficiency than 100×100 and will report it explicitly in the final results.
3. Dynamic scheduling overhead vs. benefit. Hotspot-4 may benefit from dynamic scheduling due to variable per-cell work; overhead may dominate on smaller meshes. We will quantify the crossover point.
4. Numerical stability at large Δt. The explicit Euler scheme is conditionally stable under a CFL-like bound relative to conductance and heat capacity. We have not formally derived the bound for our heterogeneous mesh; this may constrain how aggressively we can scale power to produce more dramatic buckling.
