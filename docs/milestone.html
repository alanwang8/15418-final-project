Work Completed

1. Serial Baseline: Heat Diffusion Engine

We implemented a fully functional serial simulator in C++17. The mesh is stored as a flat row-major array of Cell structs, each holding temperature, material properties (heat capacity c, conductivity k, power generation p), and a full 3D position (x, y, z). The thermal update follows the explicit finite-difference discretization: 

T_i^new = T_i + Δt · (p_i + Σ_{j∈N(i)} k_ij · (T_j − T_i)) / c_i

where k_ij = min(k_i, k_j) / dist3D(i,j) — conductance scales inversely with the 3D Euclidean distance between cells, so geometry directly affects heat flow. We use double-buffering: all reads come from the current temperature array and writes go to a separate buffer which is swapped at the end of each step. This eliminates race conditions and ensures all cells update from the same consistent snapshot. Boundary cells are held at ambient temperature (300 K) to act as heat sinks.

2. 3D Geometry Update (Buckling)

Per instructor feedback, we extended the model beyond a flat 2D plane to support out-of-plane deformation. After each thermal step, each cell's z-coordinate is updated: 

z_i = α_i · (T_i − T_ref) · base_size · Z_amplify

Hot cells rise above the chip plane; cooler cells remain near z = 0. This buckling changes the 3D distances to neighboring cells, which feeds back into the conductance weights in the subsequent timestep — creating a genuine thermomechanical coupling loop.

3. OpenMP Parallel Implementation

We parallelized both the thermal update loop and the geometry update loop using #pragma omp parallel for. We support both static and dynamic scheduling strategies, selectable at runtime. The implicit barrier at the end of each omp parallel for ensures the geometry update never runs before the thermal update completes, preserving correctness.

Thread count is controlled via CLI flag (--threads N), and all timings are measured with std::chrono::high_resolution_clock for the total run, with separate accounting for thermal vs. geometry phases.

4. Floorplan Configurations

We implemented four floorplan patterns to exercise different workload characteristics:

  • Uniform: All cache-type cells with equal power — minimal imbalance, best for baseline scaling.
  • Hotspot-1: Single ALU cluster at center with 5× power — creates one localized thermal hotspot.
  • Hotspot-4: Four ALU clusters at quadrant centers with interconnect strips — models a realistic multicore layout.
  • Random: Randomly assigned types and power multipliers — exercises non-uniform workload handling.

Preliminary Results

Correctness Validation

Key check: Serial and 8-thread OpenMP runs on the same input produce identical T_max and T_mean values (e.g., both yield T_max = 309.999 K for hotspot-1, 100×100 mesh, 100 steps). Double-buffering eliminates all race conditions.

The heatmaps below show results after 200 timesteps on a 100×100 mesh. The hotspot-1 temperature map shows the expected Gaussian-like thermal plume radiating outward from the central ALU cluster. The corresponding z-displacement map shows the central cells buckling upward as they heat — the 3D deformation creating a dome-like profile that changes neighbor distances and thus conductance in subsequent steps.

Concerns & Open Questions

Load imbalance at higher thread counts: Parallel efficiency drops from ~95% at 4 threads to ~81% at 8 threads for uniform loads, suggesting synchronization overhead or false sharing. We plan to profile with hardware counters and try cache-line-aligned data layouts.
Scalability to large meshes: We have not yet run experiments on 500×500 meshes. Time per timestep will be ~25× larger; memory bandwidth may become a bottleneck when the mesh no longer fits in L2 cache. GHC machines will clarify this.
Dynamic scheduling: We expect dynamic scheduling to outperform static for hotspot-4 (where some cells do much more work due to large temperature gradients) but underperform for uniform loads. We will compare both experimentally.
