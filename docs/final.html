with Parallel OpenMP

Results   Milestone Report   Project Home

Summary

We implemented a parallel thermo-elastic mesh simulator in C++17 with OpenMP that models heat diffusion and out-of-plane buckling on a structured processor-floorplan mesh. Each timestep (i) diffuses heat across neighboring cells using a finite-difference stencil with 3D-distance–dependent conductance, then (ii) computes the vertical displacement of each cell from thermal expansion, feeding the updated geometry back into the next thermal step — a genuine coupled thermo-mechanical feedback loop.

We parallelized the thermal update with both static and dynamic OpenMP loop scheduling, achieving up to 7.4× speedup on 8 threads (500×500 mesh, L3-cached, 92.5% parallel efficiency) and 6.2× at 100×100. A Structure-of-Arrays (SoA) + computed-neighbor optimization (v7) reduced the 500×500 hot-loop working set from ~22 MB to ~8 MB, fitting in the 16 MB L3 cache and enabling near-linear strong scaling across all four floorplan patterns. We analyzed static vs dynamic scheduling trade-offs, false sharing, cache effects, and time breakdown across mesh sizes, establishing that the thermal stencil achieves >92% parallel efficiency at 8 threads on an exclusive GHC node. The simulator also targets PSC Bridges-2 for 128-thread scaling experiments.

Background

Motivation

Modern processors suffer from non-uniform power density — ALU clusters, cache banks, and memory interfaces all dissipate heat at different rates. Accurately predicting how heat diffuses and how the die mechanically deforms under thermal stress is essential for reliability analysis, floorplanning, and package design. Simulating this at fine spatial resolution (hundreds of thousands of cells, thousands of timesteps) is compute-intensive and a natural target for parallelism.

Physics

Each cell i carries temperature Ti, material properties (heat capacity c, conductivity k, power p, thermal expansion coefficient α), and a 3D position (x, y, z) where z = 0 initially (flat chip).

Thermal update — explicit finite differences:

T_i^new = T_i + Δt · ( p_i + Σ_{j∈N(i)}  k_ij · (T_j − T_i) ) / c_i

where  k_ij = min(k_i, k_j) / dist3D(i, j)
       dist3D(i, j) = √((x_i−x_j)² + (y_i−y_j)² + (z_i−z_j)²)

Boundary cells are held at ambient temperature (300 K). The key detail is that dist3D uses the current z-positions: as cells buckle upward, they move farther from their neighbors, reducing conductance.

Geometry update — 3D thermal expansion:

z_i = α_i · (T_i − T_ref) · base_size · Z_amplify      (Z_amplify = 10)

Hot cells rise; cool cells remain flat. The updated z values feed back into dist3D in the next step. This is the coupled feedback loop: the buckling surface is not decorative — it changes the physics of the next timestep.

Double-Buffered Thermal Update (Race-Free Parallelism)

The canonical parallel pitfall for stencil codes is writing Tinew before a neighbor reads Ti. We eliminate this entirely with a separate write buffer: 

// Each parallel step:
for each cell i (parallel):
    T_buf[i] = T[i] + dt * flux(T, i) / c[i]   // reads T[], writes T_buf[]
swap(cells.T, T_buf)                             // pointer swap after barrier

Because T_buf is write-only during the parallel loop and cells.T is read-only, the entire #pragma omp parallel for region has no data dependencies between iterations. No atomics, no locks, no critical sections.

3D Conductance Feedback Loop

The key physics distinguishing our simulator from a flat 2D implementation:

  1. Step t = 0: all cells flat (z = 0), dist3D = dist2D.
  2. Thermal update computes new temperatures using 2D distances.
  3. Geometry update: hot cells rise — z_i = α_i · ΔT_i · base_size · 10.
  4. Step t+1: dist3D now includes the z-offset → reduced conductance to elevated neighbors → heat trapped in buckled region (positive feedback).
Quantitative 3D feedback effect: At 500 steps on a 500×500 hotspot-1 mesh, the 3D simulator is +0.148 K hotter than a flat-mode baseline (z ≡ 0). The divergence grows with more steps. See Fig. 3.

Optimization Iterations

v1 — Naïve single-buffer parallel for (incorrect)

First attempt applied #pragma omp parallel for to the update loop with a single cells[] array. Thread A could read Tj just as thread B was overwriting it — non-deterministic results that diverged from serial output.

v2 — Double-buffered parallel for (correct)

Introduced T_buf[] as a dedicated write target, swapping at step end. Results became deterministic and bit-identical to serial at all thread counts.

v3 — 3D geometry feedback

Added stepGeometryOMP() and updated conductance to use 3D distance. Geometry adds ~14% to runtime (one mul-add per cell, no inter-cell deps) while enabling the full thermo-mechanical coupling.

v4 — Precomputed conductances + T_flat ping-pong + NUMA-aware init

Precomputed cond_[i*4+n] = min(ki,kj)/dist3D(i,j) once at construction, eliminating sqrt from the hot loop. Extracted a dense T_flat[] float array for temperature (replacing stride-44 cells[i].T reads) with ping-pong swap via std::swap(T_flat, T_buf). Added NUMA-aware first-touch parallel init. Result: 1T runtime dropped from ~1770 ms to ~768 ms (2.3× faster per step). Added per-thread timing instrumentation.

v5 — 64-byte padded Cell (rejected)

Tried alignas(64) padding to eliminate false sharing at thread partition boundaries. Rejected: Cell struct grew from 44 → 64 bytes, expanding the cells[] working set from 11 MB to 16 MB. Extra DRAM traffic cost more than false-sharing eliminated.

v6 — Dynamic scheduling (rejected)

Tried schedule(dynamic, 64) to absorb load imbalance across hotspot patterns. Rejected: at 8T hotspot-4, dynamic was 14% slower than static (75 ms vs 66 ms). Each cell executes an identical 4-neighbor stencil — uniform arithmetic cost means static partitioning produces near-perfect balance without any dispatcher overhead.

v7 — SoA hot fields + computed neighbors + z_flat + -O3

Root cause of 500×500 scaling plateau: the 22 MB hot-loop working set exceeded GHC's 16 MB L3 cache, making the kernel DRAM-bandwidth-bound. Fix:

  • SoA extraction: added p_flat[], c_flat[], boundary_flat[] (2.25 MB total) to replace per-cell reads from the 11 MB AoS cells[] struct (20% field utilization).
  • Computed neighbors: freed neighbors[] (5 MB) after conductance precomputation; replaced indirect index lookups with inline T[i±1] / T[i±cols] arithmetic for interior cells.
  • z_flat output buffer: geometry step writes to compact z_flat[] (1 MB) instead of scattered cells[i].z (within 11 MB AoS), preventing geometry writes from evicting the thermal working set from L3 between timesteps.
  • -O3 -march=native: enables AVX2 auto-vectorization (FMA, VBLENDPS) on the stencil loop.

Working set after v7: ~8.25 MB (T_flat + T_buf + cond_ + SoA fields) vs ~22 MB before — fits entirely within GHC's 16 MB L3. On an exclusive GHC node: v7 1T ≈ 481 ms, 8T ≈ 65 ms — a 7.4× speedup (92.5% parallel efficiency), confirming that L3 residency enables near-linear strong scaling across all four floorplan patterns.

What Limited the Speedup?

1. Working set size and L3 residency (resolved by v7 SoA)

The primary bottleneck before v7 was working set size: the v4 hot-loop footprint (cells[] AoS 11 MB + neighbors[] 5 MB + T_flat+T_buf 2 MB + cond_ 4 MB ≈ 22 MB) exceeded GHC's 16 MB L3, making every thermal step DRAM-bandwidth-bound and capping scaling at ~3× at 8T.

The v7 SoA optimization resolved this: extracting the three hot fields (p_flat, c_flat, boundary_flat) into compact arrays and computing neighbor indices as i±1, i±cols eliminated the 5 MB neighbors[], reducing the working set to ~8.25 MB. A critical additional fix: the geometry loop was writing to scattered cells[i].z within the 11 MB AoS struct, evicting thermal arrays from L3 each step; redirecting to a compact z_flat[] (1 MB) preserved L3 residency across timesteps. With 8.25 MB < 16 MB L3, the 500×500 kernel is now compute-limited — achieving 7.4× at 8T (92.5% parallel efficiency).

At 100×100 (working set ~0.6 MB after v7, fits in L2), the kernel has always been compute-limited and reaches 6.2× at 8T. The slightly lower efficiency compared to 500×500 reflects the higher relative cost of OpenMP barriers at smaller grain sizes.

2. OpenMP barrier overhead

Each timestep requires a barrier (after thermal update, before geometry). For small meshes (N=100, ~0.1 ms/step), barrier overhead is a non-trivial fraction. At N=500 (~2 ms/step), the compute-to-overhead ratio improves, giving better efficiency.

Static vs Dynamic Scheduling

Main finding: static scheduling outperforms dynamic at all 1–8 thread counts across all four patterns. The thermal stencil has identical per-cell arithmetic cost; there is no genuine load imbalance from computation. Dynamic scheduling's chunk-dispatch overhead (~2–5 μs per chunk) exceeds any rebalancing benefit. Concrete numbers at 8T, hotspot-4:

  • Static: ~66 ms → speedup 7.37×
  • Dynamic: ~75 ms → speedup 6.47× (14% slower than static)

Static wins at all 1–8T because per-cell arithmetic cost is uniform (4-neighbor stencil). Dynamic's chunk-dispatch overhead (~2–5 μs per chunk) exceeds any rebalancing benefit. PSC results at 128 threads will characterize whether dynamic scheduling becomes advantageous when NUMA effects introduce genuine load imbalance across sockets.

Per-Thread Load Balance

Load balance measured across configurations:
  • 4T, static: max deviation <0.02% — essentially perfect
  • 8T, static: max deviation 0.28% — each thread's ~62 rows have uniform stencil cost
  • 16T, static: max deviation 2.5% — OS scheduling jitter, not algorithmic imbalance
Near-perfect balance at 4–8T confirms the theoretical prediction: because every cell executes the same 4-neighbor stencil, row-based static partitioning gives equal work per thread. Imbalance appears only when threads outnumber physical cores.

Time Breakdown: Thermal vs Geometry

The geometry update runs in parallel with no cell-to-cell dependencies (each cell's z depends only on its own temperature). Even though it is also parallelized, it contributes only ~14% of runtime. By Amdahl's law, optimizing the geometry step further would yield at most 14% total improvement — all optimization effort is correctly focused on the thermal update.

False Sharing & Cache Line Padding

Cell structs are 44 bytes; a 64-byte cache line holds 1.45 cells. At a static scheduling boundary, two adjacent threads write T_buf cells that may share a cache line, causing cross-core invalidations. We added an alignas(64) padded Cell build (make padded, sizeof(Cell)=64) to study this effect.

The per-thread timing data (Fig. 7a) shows <0.3% load variance at 8T with the unpadded layout, suggesting false sharing is not the dominant cost. The thermal stencil reads 5 cells per write — it is read-heavy; false sharing only affects writes. With 500 columns, only 1–2 cells per row are at a partition boundary. The padded layout eliminates this at the cost of 45% more memory per cell (64 vs 44 bytes), which can itself hurt performance by increasing the working set size. The full comparison is in results/ghc_benchmark.csv (padded=0 vs padded=1 rows).
