# Project Proposal: Dynamic Thermo-Elastic Mesh Simulation for Parallel Systems

## Summary  
We propose to build a parallel simulator of thermal diffusion on a dynamic 2D mesh, where each mesh element can deform based on its temperature (inspired by shape-memory alloys).  The mesh represents a simplified processor floorplan with heterogeneous blocks (e.g. ALUs, caches) having different thermal properties.  At each timestep, element temperatures are updated using a finite-difference heat diffusion model, and then each element’s geometry is adjusted according to its temperature.  This creates feedback: hot regions expand (changing neighbor distances) and thus alter heat flow.  We will parallelize the temperature and deformation updates across multiple CPU cores and analyze performance and scalability.  Our implementation will compare static vs dynamic mesh behavior, explore structured vs unstructured mesh representations, and measure speedups, load balance, and the overhead of mesh updates.

## Goals and Deliverables  
- **Planned (must-have) deliverables:**  
  - **Thermal simulation engine:** A working simulator that models heat diffusion on a 2D mesh.  Each element has a temperature, thermal conductivity, heat capacity, and power generation rate (to simulate workload).  We will compute temperature updates from neighbors (discretizing the heat equation) each timestep.  
  - **Temperature-dependent deformation:** Each element’s geometry (e.g. effective size or position) will change based on its temperature, mimicking a simple thermal expansion/shape-memory effect.  This will modify neighbor relationships or distances over time.  
  - **Parallel implementation:** We will implement the simulation in C/C++ with multithreading (e.g., OpenMP or pthreads), dividing elements among threads each timestep.  We will use techniques like double-buffering or barriers to ensure correct dependencies while enabling concurrency.  
  - **Performance evaluation:** We will measure execution time, speedup, and load balance.  Experiments will vary problem size (mesh resolution), use cases, and number of threads to assess scalability.  We will analyze which factors (dependency, synchronization, or mesh irregularity) limit performance.  Results will include graphs of speedup vs. threads and visualizations (e.g., heatmaps of the mesh).

  - **Mesh topology updates:** Extend the simulator so that elements can change neighbors (e.g., splitting or merging cells) as they deform, making the mesh truly unstructured.  
  - **Advanced parallel strategies:** Experiment with dynamic scheduling or work-stealing to handle load imbalance caused by nonuniform workloads. Possibly prototype a GPU version for comparison.  
  - **Enhanced visualization:** Create animations showing the evolving mesh and temperature distribution to illustrate the simulation dynamics.  
- **Contingencies:** If development falls behind, we will at least deliver the core static mesh simulator and one simple deformation model.  Topology changes and GPU implementation can be deferred.

## Background and Related Work  
Shape-memory alloys (SMAs) are materials that undergo reversible shape changes in response to temperature.  In particular, heating an SMA triggers a phase transition (martensite→austenite) that can produce large strains, effectively making the material “remember” a predefined shape【6†L150-L154】【27†L185-L190】.  In other words, an SMA inherently “senses” temperature and “acts” (deforms) accordingly【6†L150-L154】【27†L185-L190】.  Figure 1 illustrates a classic four-step SMA cycle: a wire stretched at low temperature (martensite) shortens when heated to a high temperature (austenite) and lengthens again when cooled【21†L249-L257】.  

【22†embed_image】**Figure:** Shape-memory alloy (SMA) thermomechanical cycle.  A strained SMA sample at low temperature (martensite phase) is heated to high temperature (austenite phase), causing it to contract toward its “remembered” shape【21†L249-L257】.  Cooling reverses the cycle.  This demonstrates how SMAs deform predictably with temperature.  

In our project, we will capture a simplified version of this effect: mesh cells will expand or contract in proportion to their temperature.  This thermomechanical coupling implies that heat diffusion and geometry evolve together over time.  Our idea of combining heat transfer with structural change is conceptually similar to work on SMA-based actuators and thermal engines【6†L150-L154】【27†L183-L188】.

Thermal modeling on processor floorplans is a well-studied problem in computer architecture.  For example, the HotSpot tool models a chip as a network of thermal resistances and capacitances, with each block (e.g. ALU, cache) having its own heat capacity and exchanging heat with neighbors【18†L19-L24】.  Our approach uses a spatial discretization instead: we overlay a mesh on the floorplan and solve the heat diffusion equation on it.  Heat diffusion in continuous domains is typically solved by finite-difference or finite-element methods; for instance, recent work solves the 2D heat equation using iterative finite-difference schemes on parallel hardware【16†L321-L329】.  We will follow this strategy, updating each cell’s temperature from its neighbors each timestep (akin to a convolution with a diffusion kernel).  Unlike regular grids, our mesh may be non-uniform or unstructured to reflect irregular chip block sizes.

Parallel simulations of heat diffusion often achieve significant speedups with careful scheduling【16†L321-L329】.  However, introducing mesh dynamics adds new challenges: as elements move, memory access patterns change, and workload can become very imbalanced if some regions heat up much faster than others.  We plan to investigate these issues, similar to how adaptive mesh methods require dynamic load balancing【14†L24-L28】.  

## Approach and Methodology  
Our simulator will have two main components: **thermal update** and **geometry update**.

- **Data structures:** We will represent the mesh in one of two ways.  One option is a 2D array of cells with varying “sizes” (structured but non-uniform).  A simpler implementation uses a fixed grid but assigns different thermal properties per cell.  The alternative is a true unstructured mesh where each element has an explicit list of neighbors (e.g., using adjacency lists).  We will prototype the simpler structured approach first for correctness, then extend to an unstructured representation if time allows.  
- **Thermal model:** Each element has temperature \(T_i\), thermal conductivity \(k_i\), heat capacity \(c_i\), and power generation \(p_i\).  At each time step, we update \(T_i\) by discretizing the heat equation: roughly \(T_i^{new} = T_i + \Delta t\,(p_i + \sum_{j\in N(i)} k_{ij}(T_j - T_i))\), where \(N(i)\) are neighbors and \(k_{ij}\) is a conductance factor.  This is analogous to explicit finite-difference integration【16†L321-L329】.  Boundary or heat-sink cells will be held at ambient temperature or have higher heat dissipation.  We will double-buffer the temperatures so all updates use the previous step’s values, avoiding race conditions.  
- **Geometry model:** After computing temperatures, we update the geometry.  For example, we can treat each cell as an elastic block that expands proportionally to \(T_i - T_{ref}\) (thermal expansion).  Concretely, we might increase its width/height or adjust its centroid position slightly.  This will change distances to neighbors (which will modify conductance weights \(k_{ij}\)) and in a true unstructured mesh might even change which cells are adjacent (if expansion causes overlap).  As a first step, we will simply adjust cell sizes and recompute neighbor distances; if feasible, we will then update adjacency lists for fully dynamic topology.  
- **Parallelization:** We target shared-memory multicore.  Each timestep’s computation (thermal update then geometry update) can be parallelized over cells since updates are local.  We will implement both static partitioning (each thread handles a block of cells) and dynamic scheduling (threads pull tasks from a work queue) to balance load as cells change temperature at different rates.  We will use standard techniques like OpenMP pragmas or manual pthreads.  Synchronization will occur at each time step’s end (global barrier) since the updates are inherently iterative.  Any global data (e.g. maximum temperature, heatmaps) will use atomic or reduction operations as needed.  We will experiment with different data layouts (array-of-structures vs structure-of-arrays) to improve cache locality on irregular access patterns.  

Throughout implementation, we will **iteratively refine** the design.  We expect to begin with a simple baseline (single-threaded, static mesh, modest resolution) and then profile it.  Based on profiling, we will optimize hotspots (e.g. reducing memory bottlenecks) and then introduce parallel threads.  If time allows, we will compare CPU threads vs. a prototype GPU version (e.g., CUDA kernels) to evaluate the effort vs. gain.  However, our primary focus is CPU parallelism and how mesh dynamics affect it.

## Evaluation Plan  
We will evaluate our system along two axes: correctness of simulation and parallel performance.

- **Correctness and Validation:**  Initially, we will verify that the static heat diffusion behaves as expected (e.g., a single hot cell diffuses heat in a Gaussian-like manner).  Then, we will validate the thermo-elastic effect: heating a region should cause neighboring cells to shift in predictable ways.  We will run simple test cases (such as a hot spot expanding uniformly) and visualize the results.  We will compare heat distribution patterns against the static case to ensure the coupling is reasonable (for example, check energy conservation or eventual steady states).  
- **Performance metrics:**  We will measure wall-clock runtime for fixed simulation durations.  Key metrics include **speedup** (time\(_{1\text{thread}}\)/time\(_{N\text{threads}}\)) and **parallel efficiency**.  We will run on a multicore CPU (e.g. a lab workstation or cluster node) and record execution time for various thread counts (1,2,4,8,...).  We will also track the **load balance**, e.g. per-thread execution time or work distribution, to see if some threads do much more work (due to mesh irregularity).  Another metric is the **overhead of dynamics**: we will compare a purely static mesh simulation vs. one with deformation updates to quantify the extra cost.  
- **Benchmark scenarios:** We will use multiple mesh sizes (e.g. 100×100, 500×500 cells) and different power patterns (uniform load, a few hot spots, random loads) to exercise cases from uniform to highly skewed workloads.  We will include at least one realistic-like chip floorplan (e.g. one part of chip hot, others cool) to mimic architecture scenarios.  
- **Visualization:** We will generate plots of temperature heatmaps (2D color grids) to illustrate thermal behavior, and optionally animations of the mesh moving.  For performance results, we will produce standard graphs (speedup vs. cores, time breakdown, etc.).  All results will explicitly state configurations (mesh size, time steps, compiler flags, machine used).  

This thorough evaluation (inspired by similar studies【16†L321-L329】) will allow us to identify bottlenecks.  For example, if speedup is poor, we will analyze whether it’s due to synchronization barriers, memory traffic, or inherent dependencies.  We will attempt to break down execution time into phases (compute, sync, update) and provide evidence for any conclusion.
