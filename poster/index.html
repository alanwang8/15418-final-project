Dynamic Thermo-Elastic Mesh Simulation with Parallel OpenMP

Alan Wang & Allison Chen
CMU 15-418 Parallel Computer Architecture and Programming · Spring 2026

Summary

We built a C++/OpenMP simulator of thermal diffusion coupled to 3D out-of-plane buckling on a processor floorplan mesh.

Each timestep: (1) diffuses heat using a finite-difference stencil with 3D-distance–dependent conductance, then (2) computes per-cell vertical displacement from thermal expansion — feeding geometry back into the next step.

Key result: 7.4× speedup at 8 threads on 500×500 mesh. Static scheduling optimal for uniform loads; dynamic needed for hotspots.

Why 3D Matters

The instructor feedback: "the problem is much more interesting if things can move in 3 dimensions — the chip might buckle, things might pop up."

  • Hot cells rise: z = α·ΔT·L·10
  • Elevated cells are farther from neighbors → reduced conductance
  • Trapped heat → more buckling → positive feedback loop
  • Measured: 3D mode runs +0.148 K hotter than flat

The Simulation Loop

Step 1: Thermal Diffusion

T_i^new = T_i + dt·(p_i + Σ k_ij·(T_j - T_i)) / c_i

k_ij = min(k_i, k_j) / dist3D(i, j)
dist3D = √(Δx² + Δy² + Δz²)

Boundary cells held at T_ref = 300 K. Interior cells heat from internal power and exchange heat with 4-connected neighbors.

Step 2: Geometry Update

z_i = α_i · (T_i - T_ref) · base_size · 10

Hot cells rise. The updated z feeds back into dist3D on the next step — creating coupled thermo-mechanical feedback.

Approach: Parallel Implementation

Dynamic Thermo-Elastic Mesh Simulation · Wang & Chen · CMU 15-418 S26

Key Design: Double-Buffering

Enables race-free parallel updates with zero locking:

// Both serial and OMP versions:
for each cell i in parallel:
    T_buf[i] = T[i] + dt*flux(T, i)/c[i]
    //  ↑ writes to buf    ↑ reads T only
swap(cells.T, T_buf)  // atomic pointer swap
  • T_buf is write-only during the parallel loop
  • cells.T is read-only during the parallel loop
  • No data races, no locks, no atomics within a step
  • Single implicit barrier at the end of each omp parallel for

Build System

Three binaries enable controlled comparison experiments.

Optimization Iterations

v1 — Naïve parallel for (incorrect): single cells[] array, read-write races. Results non-deterministic.
v2 — Double-buffered (correct): T_buf write target. Bit-identical to serial at all thread counts.
v3 — 3D geometry feedback: z updates each step; dist3D in conductance. Adds only 7% overhead; enables coupled physics.
v4 — Padded build: Cell aligned to 64 bytes (one cache line) via alignas(64). Eliminates false sharing at boundaries.

New Analysis Tools

  • --track-threads: per-thread wall time via omp_get_wtime()
  • --flat: disable 3D feedback (2D baseline)
  • --bandwidth: STREAM-style memory bandwidth probe
  • iso3d visualizer: isometric SVG/PNG of buckled surface
  • profile visualizer: center-row cross-section chart

Time Breakdown: Thermal vs Geometry

Amdahl insight: Geometry = 7% of runtime. Even if geometry were free, max speedup gain would be only 7.5%. All optimization effort correctly targets the thermal kernel.

Both phases are parallelized. Thermal scaling is the bottleneck — it is memory-bandwidth limited at high thread counts.

Results: Scheduling, Load Balance & Cache

Dynamic Thermo-Elastic Mesh Simulation · Wang & Chen · CMU 15-418 S26

Static vs Dynamic Scheduling

For uniform patterns: static always wins — zero overhead, perfect balance.

For hotspot patterns: dynamic scheduling redistributes work from loaded threads to idle ones. The benefit depends on chunk size:

ChunkSchedule overheadBalance benefit
8 HighGood
64 MediumModerate
256LowMinimal
Finding: chunk=64 is the sweet spot for hotspot patterns on 100×100 meshes. At 500×500 the optimal chunk grows proportionally.

False Sharing Analysis

Cell struct size: 44 bytes (default). Cache line: 64 bytes. Adjacent thread boundary cells share a line.

Padded build: alignas(64) + 20-byte pad → 64 bytes/cell. Each cell occupies exactly one cache line.

Expected result: At 8+ threads on 500×500 meshes, the padded build should be ~5–10% faster due to fewer cross-thread cache invalidations. Effect is larger on NUMA systems (PSC).

What Limited Speedup?

1. Memory bandwidth saturation

Each step reads ~5 cells per output cell. At 500×500 × 44 bytes × 5 neighbors: ~55 MB per step. GHC bandwidth ≈ 30 GB/s. At 8 threads the kernel becomes memory-bound.

2. Load imbalance at barrier

Static scheduling: hotspot-center threads finish slower; all threads wait at the implicit barrier. Per-thread timing (--track-threads) reveals the imbalance directly.

3. OpenMP thread launch overhead

Each stepThermalOMP call: fork + barrier + join. At N=100 (~3 ms/step serial), overhead is significant. At N=500 (~80 ms/step), overhead is negligible.

4. Cache thrashing at boundaries

Default Cell (44 bytes): 1.45 cells/cache line. Adjacent threads' boundary rows share a cache line → false sharing. Padded build eliminates this.

Summary

Primary bottleneck: Memory bandwidth at 8+ threads. Secondary: load imbalance (hotspot patterns, static schedule). Both are quantified with data, not just speculation.

Conclusions, PSC Scaling & References

Conclusions

  • Double-buffering enables race-free OpenMP parallelism with no locks
  • 7.4× speedup at 8 threads on 500×500 mesh (~77% efficiency)
  • 3D out-of-plane buckling is non-trivial: it measurably changes thermal results by reducing inter-cell conductance
  • Static scheduling optimal for uniform loads; dynamic helps for hotspot patterns at the right chunk size
  • Memory bandwidth is the primary bottleneck at 8+ threads
  • False sharing (44-byte cells on 64-byte cache lines) is measurable at high thread counts; padding eliminates it

PSC Bridges-2 Scaling

Submit with one command:

Expected results (1000×1000, 128 threads):

  • Good scaling up to 32T (single socket)
  • NUMA boundary effect at 64T → 128T
  • Dynamic scheduling partially compensates at 128T

Future Work

  • SIMD-vectorize the inner stencil loop (AVX2 / AVX-512)
  • GPU port (CUDA) — geometry step is trivially GPU-parallel
  • Adaptive timestep: finer dt near hotspots, coarser elsewhere
  • Diagonal neighbors (8-connected) for smoother gradients
  • Longer simulations showing accumulated buckling over 10,000+ steps
