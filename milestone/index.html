Summary

This milestone summarizes a C++/OpenMP simulator of thermal diffusion coupled to out-of-plane deformation (vertical warping) on a structured processor floorplan mesh. Each timestep applies an explicit thermal update with 3D distance-dependent conductance (kij = min(ki,kj) / dist3D), then updates each cell’s z coordinate from thermal expansion so geometry feeds back into the next step. The implementation uses a race-free double-buffered temperature update, static and dynamic OpenMP loop scheduling, and a small CLI for threads, mesh size, timestep count, floorplan pattern, and optional field dumps and timing output. We measure strong scaling (including on GHC machines) and produce heatmaps and speedup plots from scripted sweeps and lightweight post-processing.

Technical approach

Mesh

The floorplan is a grid of square cells in row-major order. Each cell precomputes indices of up to four edge neighbors (no diagonals). A Cell holds temperature T, capacity c, conductivity k, power p, expansion α, positions x, y, z, base_size, type (ALU, cache, memory, interconnect), and a boundary flag. Boundary cells are held at ambient temperature (300 K).

Thermal update

Interior cells use an explicit update with reads from the current field and writes to a separate buffer; buffers swap each step (double-buffering):

T_i^new = T_i + Δt · (p_i + Σ_{j∈N(i)} k_ij · (T_j − T_i)) / c_i

Geometry and coupling

After the thermal phase,

z_i = α_i · (T_i − T_ref) · base_size · Z_amplify

with Z_amplify = 10 in code. Conductance uses 3D separation between cell centers, so updated z changes dist3D and kij on the following step.

Parallelism

We first validated a serial simulator on small meshes, then added OpenMP #pragma omp parallel for on stepThermal and stepGeometry. An implicit barrier between the two phases ensures geometry sees a consistent post-thermal T. Scheduling is selectable (static vs. dynamic with chunk size). Wall-clock timing uses chrono with separate thermal and geometry accumulators.

Simulation workflow and test inputs

Each run allocates the mesh with fixed cell spacing, sets ambient T and flat z = 0, then calls initFloorplan to assign materials and power. The main loop times stepThermal and stepGeometry and records Tmax and Tmean. Optional flags control mesh size (--size or --rows/--cols), timesteps (--steps, --dt), threads and schedule, --serial for reference runs, --output for appended timing rows, and --heatmap for a final per-cell snapshot (row, column, T, z, type) for plots.

Floorplan patterns (selected via CLI) are procedural scenarios, not hand-authored input files:

  • Uniform: homogeneous cache-like tiles—nearly uniform T and flat z baseline.
  • Hotspot-1: cache baseline with a central ALU patch (~10% mesh extent per dimension), high power multiplier—single plume and localized warping.
  • Hotspot-4: memory strip on one edge, four ALU clusters near quadrant centers, interconnect along mid-row/mid-column corridors between quadrants.
  • Random: weighted random types and per-cell power jitter; RNG seed (--seed) keeps layouts reproducible across serial and parallel runs.

Boundary power is zeroed so edges act as fixed-temperature sinks.

Preliminary results

Uniform baseline

After convergence, temperature is nearly uniform at ambient scale and z stays nearly flat—sanity check before hotspot gradients.

Correctness

Check: Serial and multi-threaded runs on identical inputs agree on the final temperature field for the cases we checked. For Hotspot-1 under the benchmark settings, aggregated runs give Tmax = 320.001 K and Tmean ≈ 301.47 K. Double-buffering avoids write races: threads read the previous step and write only into the next buffer before commit.

Observation: Uniform scaling reaches ~8× at 8 threads on this mesh; efficiency above 100% at some points suggests cache effects (smaller per-thread working sets). Hotspot-1 scales more slowly (6.2× at 8 threads): equal cell partitioning still allows varying work per cell (gradients, conductance), mild imbalance, and barrier wait. Hypotheses to test on GHC include barrier cost, memory bandwidth at 500×500, false sharing, and dynamic scheduling tradeoffs.

Concerns and open questions

  1. Load imbalance and false sharing. Efficiency drops at 8 threads for Hotspot-1; we plan profiling (e.g. perf on GHC) and layout experiments (alignment, padding, or SoA for hot fields).
  2. Memory bandwidth at 500×500. Larger meshes stress caches; we expect lower parallel efficiency than 100×100 and will report it explicitly.
  3. Dynamic scheduling cost. Hotspot-4 may benefit from dynamic scheduling; overhead may dominate on small meshes—we will quantify crossover.
